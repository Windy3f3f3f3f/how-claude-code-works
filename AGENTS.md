# 简明版文档写作规范

> 本规范用于把 `docs/` 下的硬核原版章节改写成 `docs-easy/` 下的简明版，面向**想理解原理但不读源码的读者**。

## 一、整体定位

- **面向读者**：想理解 Claude Code 原理、但不打算读源码的人。
- **与原版关系**：大纲完全对齐原文（小节编号、顺序、标题一致），方便对照阅读。砍掉血肉、保留骨架——重点是"为什么这么设计"，不是"代码怎么实现"。
- **输出位置**：新建 `docs-easy/` 目录，不动原 `docs/`。
- **篇幅**：约原文的 1/3。

## 二、贯穿例子的设计规范（核心）

**一个精心设计的连续例子贯穿全章，背景只交代一次。** 不允许每个知识点换一个新例子，更不允许用额外比喻（如"办公桌""体重秤"）来解释——所有概念都要落到这个例子的真实场景里讲。

### 例子设计原则

1. **足够真实**：一个真实的编码会话，读者能代入。
2. **足够长**：能自然触发该章涉及的所有机制（如压缩、缓存失效、PTL 兜底等）。
3. **背景只交代一次**：开头交代项目设定，后面每节用"接着上面的任务"这类短句接上，读者不用重新入戏。
4. **知识点自然涌现**：每个机制都是任务走到那一步才冒出来的，不为讲知识点硬造场景。
5. **数字落到例子里**：比如"日志几万行""压缩前 18 万 token"，给读者实感。

### 例子设计流程（每章开工前先做）

1. 读完整章原版，列出大纲（所有小节）。
2. 设计一个能覆盖全部小节的故事线，画一张"故事阶段 → 知识点"映射表给用户确认。
3. 确认技术栈和任务场景（问用户，贴近其日常最佳）。
4. 例子定稿后才开始写正文。

> 第 3 章的例子：接手一个 Spring Boot 项目（Maven、有 CLAUDE.md、装了 GitHub MCP、根目录有 application.log），让 Claude Code 加一个用户登录接口。贯穿 11 个小节。

## 三、语言表达规范

### 1. 禁止"不是X而是Y"句式

不要先否定、再给结论。**直接正面表达想说的**。

| ❌ 禁止 | ✅ 改成 |
|--------|--------|
| 不是X，而是Y | 直接说Y（或"X改用Y"） |
| 而非 | 不用 / 改用 |
| 与其A不如B | 趁B，比A快得多 |
| 不是随便排的 | 有讲究 |
| 不是用户原话 | 与用户输入无关 |
| 不截断，而是落盘 | 结果落盘，上下文只留引用 |

> 写完每章必须 grep 复查：`而是`、`而非`、`与其.*不如`、`不是.*是` 等模式，清零。

### 2. 禁止引入额外比喻

既然已有贯穿例子，**所有概念围绕例子本身的真实场景解释**，不要再造"办公桌""体重秤""抽屉"这类第二层比喻。例外：原文已有的、且已成为行业通用术语的类比（如"缓存命中""熔断器"）可保留。

### 3. 每节结构统一

点题句直接加粗，不写显式点题标签 → 用例子场景解释 → 关键设计点（2-3 条）→ 简短小结。原文"源码→逐行注释→设计动机"的学术写法，改成"先讲结论再讲为什么"。

## 四、内容取舍规范

### 砍掉

- TypeScript 函数签名、`file:line` 引用、`src/...` 路径
- 大段源码块及逐行注释
- 精确到字符数的常量表（只留最关键的几个数字）

### 保留并强化

- **关键数字**：能形成直觉的（200K 窗口、5 分钟 TTL、autocompact ~90% 触发）
- **流程图/mermaid 图**：直观，保留并按需简化
- **"为什么这么设计"**：设计动机是理解原理的关键，提到显眼位置
- **设计权衡**：牺牲了什么、换来什么

## 五、每章固定结构

1. **开头**：一个加粗的本章核心结论 +「读完你能知道什么」——帮读者判断要不要读。正文只保留结论本身，不写显式点题标签。
2. **贯穿例子设定**：交代项目背景、任务场景（只此一次）。
3. **正文**：按原文大纲逐节展开，每节用例子接续。
4. **实践指南 / 设计洞察**：原文如有对普通用户有用的实践章节，保留并口语化。
5. **结尾**：对比阅读指引——指向原版 `docs/` 对应章节，说明简明版与原版的分工。

## 六、推进流程

1. 选定一章，先**完整读完原版**，理清大纲。
2. **设计贯穿例子**，把"故事阶段→知识点"映射表给用户确认（含技术栈/场景询问）。
3. 例子定稿后写正文，遵循上述规范。
4. 写完 grep 复查违规句式（而是/而非/比喻词）。
5. 给用户过目，满意后按同标准推进下一章。

## 七、参考资料

讨论或修订文档时，必要时可参考以下两处：

- **原版文章**：`docs/` 下对应章节——查证某个知识点的原始表述、精确数字、设计动机。
- **Claude Code 源码**：`ref-code/claude-code/` 下——当用户就某处细节追问、或原版表述存疑时，回源码核实实现，避免以讹传讹。

> 这通常发生在用户跟你讨论已写出的新文档时——用户提出质疑或追问，你再去回查原版/源码确认，而不是写作阶段就逐条核对源码（那会拖慢节奏、也偏离简明版定位）。

# 第 3 章：上下文工程（简明版）

> **本章一句话**：模型的决策质量完全取决于它"看到"了什么——上下文工程就是决定把什么放进模型视野、什么时候把旧的挪出去、又怎么挪得不破坏缓存的一套管理系统。

> **读完你能知道**：一次请求里到底装了什么；对话变长后系统怎么压缩、分几级；前缀缓存为什么是"带着镣铐跳舞"；以及你日常哪些习惯能让 Claude Code 更快更省。

---

## 贯穿全章的例子

后面所有知识点都会围绕同一个任务讲，背景只交代这一次。

你接手了一个别人的 **Spring Boot** 项目（Maven 构建，REST API），让 Claude Code 给它加一个**用户登录接口**。这个项目：

- 有一份 `CLAUDE.md`，写着「测试用 JUnit 5 + Mockito，Controller 放在 `controller/` 包下，Service 层不直接碰数据库，走 Repository」
- 你装了一个 GitHub MCP（偶尔查 issue 用）
- 项目根目录有个 `application.log`，几万行运行日志

开工。

---

## 为什么上下文工程如此重要？

模型有个固定大小的"视野窗口"（Claude 最大约 100 万 token）。但你这一通改代码——读十几个文件、跑好几轮测试、看一堆报错——产生的原始文本轻轻松松就破百万 token。

所以系统必须做取舍：**哪些留在视野里，哪些压缩掉**。取舍不当，模型会忘记刚改了哪个文件、重复读已经看过的代码、甚至前后自相矛盾。

回到登录任务，这件事会贯穿整场会话：Claude 要决定把 `AuthController.java`、`pom.xml`、测试报错这些留在模型视野里，把过时的旧工具结果挪出去，压缩后还要把最近读过的文件再捞回来。这三件事分别叫**上下文构建、压缩、恢复**，正是上下文工程要解决的。

但还有一层更狠的约束：Claude Code 每次请求，光系统提示词和工具定义就占 **5 万~10 万 token**。为了不每次都从头算，它依赖服务端的**前缀缓存**（服务端记住上次算过的前缀，下次只算新增部分）。

而这个缓存有个残酷规矩：**前缀必须逐字节完全一致才能命中**。改一个字、换一个工具的顺序，整个 5~10 万 token 的前缀全部失效、全部重算。

这就是全章反复出现的那股张力——每个设计决策都得同时满足两个目标：**给模型最好的上下文**，**又不打破缓存**。很多看似"过度设计"的机制，背后都是这股张力在驱动。

## 3.1 上下文构建全景

每次调 API，模型都是从零开始——它没有跨请求的记忆，只看得到这次请求里带的东西。所以 Claude Code 每次都得在请求前，把模型需要的一切组装好。三大支柱：

1. **系统提示词**：定义模型"是谁、该怎么做"。最稳定，跨请求基本不变 → 缓存的最佳候选。
2. **系统/用户上下文**：环境信息（git 状态、平台）和项目知识（CLAUDE.md）。每会话算一次。
3. **消息历史**：你说的、模型回的、工具调用和结果——变化最快、占空间最大。

回到我们的登录任务。你说"加个登录接口"，Claude 组装的第一次请求大致长这样：

```
┌───────────────────────────────────────────────────────┐
│ system（系统提示词）                                     │
│  - 归属头（谁发的请求）                                  │
│  - 核心指令 + 工具描述 + 安全规则   ← 所有用户完全相同    │
│  ── 静态/动态边界 ──                                    │
│  - 输出风格、语言偏好等             ← 因你而异            │
│                                                         │
│ tools（工具清单）                                       │
│  - Read, Edit, Bash, Grep, Write, Glob...              │
│  - 你的 GitHub MCP 工具                                 │
│                                                         │
│ messages（消息历史）                                    │
│  [User] <system-reminder> CLAUDE.md 内容 + 今天日期     │
│  [User] "加个登录接口"                                   │
└───────────────────────────────────────────────────────┘
```

注意几个反直觉的设计，后面会反复用到：

- **CLAUDE.md 和日期塞在消息数组的第一条，没放进系统提示词**。因为 CLAUDE.md 因项目而异，放系统提示词里会拖垮缓存共享率。
- **记忆、技能、MCP 指令作为 `<system-reminder>` 附件消息按需注入，不进系统提示词**——这样只在内容变化时才加新的，碰不到系统提示词的缓存。
- **工具清单最后一个工具打了缓存断点标记**，服务端缓存到这为止。Advisor 之类的可选工具放在断点之后，开关它不影响前面的缓存。

**一轮对话怎么改变上下文**（动态视角）：

```
每一轮开始
   ↓
① 压缩检查（五级流水线逐级筛查）  ← 对话长了才真正干活
   ↓
② 组装请求（system + tools + messages）
   ↓
③ 发给 API，流式接收模型回复（可能含工具调用）
   ↓
④ 有工具调用？→ 执行工具，收结果
   ↓
⑤ 收集附件消息（记忆预取、技能增量、MCP 指令增量）
   ↓
⑥ 拼回消息数组：messages += [模型回复, 工具结果, 附件]
   ↓
回到顶部，进入下一轮
```

核心：上下文是"活的"。每一轮消息数组都在长（新回复 + 工具结果 + 附件），而压缩机制在每轮开头压住增速。系统提示词和工具清单整场几乎不变——这正是它们能被高效缓存的原因。

## 3.2 系统提示词的构建

系统提示词是最稳的部分——定义模型"是谁、怎么做"。正因为稳，它是缓存的最大受益者。Claude Code 在"稳"和"活"之间做了精心切分。

### 静态/动态边界：一句话切开两半

系统提示词里有个哨兵字符串 `__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__`，把提示词切成两半：

- **边界之前**：核心指令、工具描述、安全规则——**全球所有用户的所有会话完全相同**
- **边界之后**：输出风格、语言偏好、MCP 指令等——**因你而异**

为什么要切？因为缓存可以分级共享。边界之前的静态部分能用 `scope: 'global'`——**全球几百万用户共享同一份缓存**。边界之后的只能 `scope: 'org'`（组织内）或不缓存。不切的话，整个系统提示词只能做 org 级，海量完全相同的内容各存一份，纯浪费。

但这个"最优策略"有前提。决策树：

| 场景 | 静态内容缓存 | 动态内容缓存 | 共享范围 |
|------|------------|------------|---------|
| 你装了会实际渲染进工具数组的 MCP 工具 | org | org | 组织内 |
| 没装 MCP + 支持全局缓存 | **global** | 不缓存 | **全球** |
| 都不满足 | org | org | 组织内 |

> 注意第一行——这正是我们例子里你装了 GitHub MCP 的隐患。**只要 MCP 工具进了工具数组，缓存就从全球共享降级到组织内**，你享受不到那几百万人的共享缓存。后面 3.6 会详细讲，以及 Tool Search 的延迟加载怎么补救。

### 系统上下文：git 快照

回到登录任务。Claude 第一次请求时，会顺手收集你的 git 状态——当前分支、主分支、最近 5 条提交、`git status`（未提交的改动）。这些一次性算好，整场会话不再更新。

值得注意两点：
- **5 个 git 命令并行跑**，大仓库能省几百毫秒
- **`git status` 截断到 2000 字符**，并在末尾明确告诉模型"这是会话开始时的快照，对话中不会更新"——防止模型后续"幻觉"出实时的 git 状态

### 用户上下文：CLAUDE.md 怎么被发现

你项目里那份"测试用 JUnit 5、Controller 放 controller 包下"的 CLAUDE.md，Claude 怎么找到的？发现过程比想象复杂：

1. 先看 MDM 策略文件（企业统一指令）
2. 再看 `~/.claude/CLAUDE.md`（你的全局配置）
3. 从当前目录**向上逐层遍历**，找每层的 `CLAUDE.md` / `.claude/CLAUDE.md` / `.claude/rules/*.md`
4. 再看 `CLAUDE.local.md`（不进 git 的个人指令）
5. 最后看 `--add-dir` 指定的额外目录

**就近原则**：离当前目录越近的文件越后加载，优先级越高。子目录的规则能覆盖根目录的。而且模型对上下文末尾的内容更敏感（近因效应），后加载的规则在模型"注意力"里权重更大。

CLAUDE.md 还支持 `@./other-rules.md` 这样的引用语法，被引用的文件插到引用者**之前**，并防循环引用。

### 注入顺序

系统上下文**追加**在系统提示词后面，用户上下文（CLAUDE.md、日期）**前置**在消息数组开头。这个顺序有讲究：最稳定的系统提示词在最前，最利于缓存命中；可能变动的用户上下文放消息里，碰不到系统提示词的缓存。

## 3.3 消息历史管理

Claude 发历史给 API 前要先"规范化"——内部消息格式和 API 要求的不完全一致。

我们的登录任务走到这儿：Claude 刚 grep 完路由、读了 `pom.xml`、准备写 `AuthController`。这几轮操作产生的消息，发出去前要过一道 `normalizeMessagesForAPI` 关卡，解决 7 类问题：

| # | 问题 | 不处理会怎样 |
|---|------|------------|
| 1 | 附件消息位置乱了 | API 看到孤立的图片，不知道关联哪条 |
| 2 | 内部临时消息（isVirtual）混进来了 | 它们只为 UI 展示，进了 API 会污染上下文 |
| 3 | 之前有"PDF 太大"之类的报错 | 同一个大文件每次请求都重发、每次都报错 |
| 4 | 内部跟踪标记（tool_reference 等） | API 不认识，直接返回格式错误 |
| 5 | thinking 块发给了不支持的模型 | API 直接 400 |
| 6 | 一个响应被流式拆成多条同 ID 消息 | 违反"用户/助手交替"规则 |
| 7 | tool_use 和 tool_result 没配对（崩溃恢复后常见） | API 报错，会话几乎必然挂 |

一句话：**无论内部状态多乱，发给 API 的永远是合法的消息序列**。这是防御层——尤其会话崩溃恢复、压缩、中途打断后，配对关系很容易破坏，这层规范化是能续命的。

> **压缩边界**：一旦 autocompact 发生过，消息列表里会插一个 `compact_boundary` 标记，之后请求只发标记之后的——历史被"翻篇"了。

## 3.4 五级压缩流水线

这是上下文管理的核心。对话越来越长、token 不断涨，五级流水线逐级启动。设计哲学是**渐进式**——先用最便宜的手段腾地方，不够再上更重的武器。

继续我们的登录任务。你已经让 Claude 改了十几个文件、跑了好几轮测试，消息开始膨胀。五级流水线逐级上阵：

```
消息列表
  ↓
Level 1  Tool Result 预算裁剪   ← application.log 几万行落盘，只留 2KB 预览
  ↓
Level 2  History Snip           ← 移除早期冗余消息
  ↓
Level 3  Microcompact           ← 清掉半小时前读过的旧文件结果
  ↓
Level 4  Context Collapse       ← 把早期消息折叠成摘要（只读视图，不动原件）
  ↓
Level 5  Autocompact            ← fork 子 agent 把整段对话浓缩成摘要
```

为什么按这个顺序？每一级都比上一级"更重"——更费算力、或丢更多细节。从免费的本地操作一路升级到要调 API 的全量摘要。

### Level 1：Tool Result 预算裁剪

Claude 决定先看一眼运行日志，执行了 `cat application.log`——几万行哗一下涌进来。单次工具结果就可能撑爆上下文。

处理方式很巧：**结果落盘，上下文里只留引用**。超过 5 万字符的输出写到磁盘文件，上下文里留一个引用 + 2KB 预览：

```
<persisted-output>
输出太大（2.3MB）。完整内容存到：/tmp/.../tool-results/toolu_xxx.txt
预览（前 2KB）：
[开头几行日志]
...
</persisted-output>
```

为什么不直接截断？截断 = 数据永久丢失。万一模型后来在第 5000 行发现 bug 的线索，它再也看不到了。落盘则保留了完整数据——模型随时能用 Read 工具把磁盘文件读回来。2KB 预览够它判断"要不要看全文"。

> 零延迟、零成本——纯本地操作，不调 API。

### Level 2：History Snip

直接从消息列表里移除冗余部分，释放量很大，可能让后面的重压缩根本不用触发。

### Level 3：Microcompact（最精巧的一级）

目标：清理"30 分钟前读的文件"那种已经没用的旧工具结果。妙在它有**两条完全不同的路径**，看缓存状态选哪条——这正是我们例子里"你一直在改代码" vs "你去喝咖啡回来"两个时刻的分野。

**路径 A：缓存已冷（你去喝咖啡了）**

你盯着屏幕改代码改了半小时，然后离开 8 分钟去喝咖啡。服务端缓存默认 5 分钟过期，你回来时它已经"冷了"——无论怎么做都得重新上传完整前缀。

既然横竖要重建，Microcompact 就**直接改消息内容**：把旧的 FileRead/Bash/Grep 结果替换成 `[Old tool result content cleared]`，只保留最近 N 个。反正缓存要重建，改了不额外亏。

**路径 B：缓存还热（你一直没走）**

你没离开，一直在活跃对话。这时缓存是热的——如果直接改消息内容，缓存键一变，**10 万+ token 的前缀全部失效**，全得重算重传。代价巨大。

所以这条路**完全不碰本地消息**，改用一个 API 级机制：在工具结果块上加个 `cache_reference`（等于工具调用 ID），再发个 `cache_edits` 指令告诉服务端"在缓存里把这些位置的内容就地删掉"。客户端免重传，服务端免重算前缀——既清了旧内容腾了空间，又保住了缓存热度。

| | 路径 A（缓存冷） | 路径 B（缓存热） |
|---|---|---|
| 触发 | 离开超时（你喝咖啡去了） | 工具结果数量超阈值（你一直没走） |
| 操作 | 直接改消息内容 | `cache_edits` 让服务端就地删 |
| 对缓存 | 本来就要重建，无额外损失 | 保住热度，避免重传 |
| 调 API | 不调 | 不调（搭下次正常请求的便车） |

两条路互斥：时间触发优先。缓存都冷了，用 `cache_edits` 也没意义。

> 可压缩的工具类型：FileRead、Bash、Grep、Glob、WebSearch、WebFetch、FileEdit、FileWrite。

### Level 4：Context Collapse（折叠，不动原件）

像数据库的 View：底层表（消息数组）不变，但查询时（发 API 时）看到的是折叠后的视图——早期不重要消息被替换成摘要。

关键特性：**不修改原始消息**，所以可逆，能跨轮持久化，需要时能回退。

阈值上它卡得很巧：在利用率约 **90%** 时提交折叠、约 **95%** 阻塞派生子任务；而 Autocompact 在约 **93%** 触发——刚好夹在中间。源码注释直说"autocompact 会和 collapse 抢、且通常赢，把 collapse 正要保的细粒度上下文炸掉"。所以**当 Context Collapse 活跃时，Autocompact 被抑制**。

### Level 5：Autocompact（最后的手段）

前面都压不住，就 fork 一个子 agent，把整段对话浓缩成摘要，原始消息被摘要替换——**不可逆**。

**触发条件**（5 个得同时满足）：
1. 排除压缩 agent 自己（防死循环）
2. 三个开关都没禁用（环境变量 / 配置）
3. 不在"反应式压缩"模式
4. Context Collapse 没在活跃（否则被抑制）
5. token 超阈值

**阈值**：`effectiveWindow - 13000`。`effectiveWindow = 上下文窗口 - 摘要输出预留(最多 2 万)`。以 20 万窗口算，触发点约在 **83.5%~89.5%**。13K buffer 保证触发时还有空间跑完当前工具和生成摘要。

**压缩提示词的巧思**：让子 agent 先写 `<analysis>`（按时间顺序分析每条消息）再写 `<summary>`（9 个标准化部分：主请求、技术概念、文件与代码、错误与修复……待办、当前工作、下一步）。但**最终只留 summary 进上下文，analysis 被丢弃**——经典的"先推理再总结，丢推理保结论"，质量远超直接摘要，又不浪费 token。

**压缩后恢复**（防止模型"失忆"）：
- 恢复最近读过的 **5 个文件**（每个 ≤5K token）
- 恢复已激活的**技能**（总预算 25K）
- 重新通告**延迟工具、Agent 列表、MCP 指令**等增量（压缩把它们吃了）
- 重置 Context Collapse 状态

没有这套恢复，模型压缩后会忘记刚改了哪些文件，后续可能重复读或改出不一致。

**熔断器**：连续 3 次压缩失败就停手——上下文不可恢复地超限。这来自真实事故：曾有 1279 个会话连续失败超 50 次（最高 3272 次），一天浪费约 25 万次 API 调用。

## 3.5 Token 预算管理

### 估算算法：拿上次 API 报的数当锚点

Claude Code 需要实时知道"现在上下文多少 token 了"来决定要不要压缩。但它**绝不为此调 API**——网络延迟会拖垮决策。

回到登录任务：Claude 刚跑完 `mvn test`，API 在响应里报了 `usage`，说这次输入精确用了 9.5 万 token。紧接着 Claude 又读了一份测试报错文件——系统直接拿这 9.5 万当锚点，只对这条新增的报错文件做粗估（字符数 ÷ 4，JSON 类内容 ÷ 2 更保守），加起来就是当前总量。

算法逻辑：从消息末尾往前找最近一条带 `usage` 的，拿这个服务端精确值当锚点，只估锚点之后的新消息。比纯客户端估算准得多（误差从 30%+ 降到通常 <5%），又不要额外 API 调用。

### 输出 token 预留

不同模型默认 `max_output_tokens` 不同（Opus 4.6 是 6.4 万，Sonnet 4.6 是 3.2 万，老模型 8 千）。但默认其实只用 **8 千**——因为统计上 99% 的请求输出不到 5 千，声明 3.2 万会让服务端过度预留计算资源。真不够时系统自动升到 6.4 万并重试（MOT 恢复机制）。

### 跨压缩结转

每次压缩前捕获"上次响应的上下文 token 数"，压缩后从剩余预算里扣除。因为压缩后服务端只看到摘要、不知道压缩前上下文多大——这个结转告诉服务端"这些 token 已经花掉了"。

## 3.6 前缀缓存策略

开篇说的"镣铐"现在细看。每次请求服务端都要对输入做 KV Cache 计算（transformer 注意力的底层活），输入可能 10~20 万 token——每次从头算延迟和成本都受不了。前缀缓存让服务端记住上次算的，下次前缀一致就复用，只算新增。但**前缀必须逐字节一致**才能复用。

### 三层缓存链

Claude Code 不只缓存系统提示词，三个层次都设了缓存断点，串成一条链：

```
请求 N：
[system ←断点] [tools ←断点] [msg1 msg2 msg3 ←断点]
  ~~~~~命中~~~~~  ~~~~~命中~~~~~  ~~~~~命中~~~~~

请求 N+1：
[system ←断点] [tools ←断点] [msg1 msg2 msg3 ←断点] [msg4 msg5 ←断点]
  ~~~~~命中~~~~~  ~~~~~命中~~~~~  ~~~~~命中~~~~~           ↑只有这要算
```

- **断点 1（系统提示词）**：在静态/动态边界处标记，核心指令部分跨用户共享
- **断点 2（工具数组）**：最后一个常规工具标记，advisor 之类可选工具放断点后
- **断点 3（消息数组）**：最后一条消息标记，历史消息全在缓存前缀内

三断点串起来：一个跑 20 轮的对话，第 21 轮只需处理最新的用户消息和工具结果，前面所有积累全命中缓存。这是 Claude Code 低延迟的核心。

### 四层防御：护住这条链

全链路缓存收益大，失效代价也大——一次意外断裂可能让 10 万+ token 全重算。四层防御：

**第一层：系统提示词分割**（3.2 讲过）。把通用/专属切开，最大化共享。但你装的 GitHub MCP 会让整条降级——MCP 工具 schema 因人而异。

**第二层：会话级锁存**。防两类隐蔽的"中途翻转"：
- *TTL 翻转*：缓存过期时间普通用户 5 分钟、付费用户 1 小时，但 TTL 本身也是缓存键的一部分。如果对话到第 10 轮你用量超标被从"1 小时"降级"5 分钟"，缓存键变了，前面全废。解法：**会话开始时锁定资格，中途不变**。
- *请求头翻转*：beta 功能（fast-mode 等）通过 HTTP 头启用，header 也影响缓存键。feature flag 随时可能在服务端切换——你正编着码，flag 关了 fast-mode，下次请求少个 header，5~7 万 token 缓存立刻作废。解法：**一旦某个 header 首次发出，整场会话持续发**（粘性锁存），即使触发它的 flag 已关。

两个锁存共享一个重置点：`/clear` 和 `/compact`——这俩本就要重建上下文，缓存必重建，锁存没保护必要。

**第三层：工具与消息的排列**。可选工具（advisor）放断点后，开关不影响前面缓存；MCP 工具走延迟加载（Tool Search 调用前不进工具数组），减少因用户特有工具导致的缓存差异；缓存热时的 Microcompact 用 `cache_edits` 就地删（见 Level 3 路径 B），避免改本地消息——消息层缓存和压缩机制的精妙配合。

**第四层：缓存断裂检测**（诊断安全网）。前三层保证缓存"应该"稳，但"应该"和"实际"有差距。系统在每次 API 调用前后做快照对比：调用后若 `cache_read_input_tokens` 比上次降超 5% 且 2000 token，判为断裂，自动归因到三种原因——TTL 过期 / 客户端变更（能定位到具体哪个工具 schema 变了）/ 服务端驱逐。这套检测形成改进闭环：第二层的锁存机制正是先靠它发现特定 header 翻转导致的断裂，才被开发出来的。

> 小结：system、tools、messages 三层断点串成完整缓存链；分割、锁存、排列、检测四层防御护住它。最终效果——无论对话到第几轮，每次只算最新增量，前面所有上下文由 KV Cache 免费提供。

## 3.7 `<system-reminder>` 注入机制

Claude Code 要在对话各处塞系统级信息（延迟工具列表、记忆内容、安全提醒），但直接插有个麻烦：**模型可能误以为是你说的**。

`<system-reminder>` 标签就是解决这个的统一通道。系统提示词里明确告诉模型：这些标签里的是系统自动加的元数据，与用户输入无关。

回到登录任务，你会在这些地方碰到它：
- 第一条消息里包着 CLAUDE.md 内容和今天日期
- Claude 读完一个空文件时，结果里附带 `Warning: file exists but is empty`
- 记忆预取结果、技能列表、MCP 指令增量，都作为附件消息裹在 `<system-reminder>` 里注入

为什么用 XML 标签？它造出清晰的语义边界——模型经训练知道这是系统注入的元数据。这样系统能在**任意位置**塞上下文（工具结果后、消息之间）而不混淆"谁在说话"。

## 3.8 记忆预取

Claude 生成回复的同时，会**并行**搜一遍本项目的自动记忆目录（`~/.claude/projects/<项目路径转义>/memory/`，同一 git 仓库的所有 worktree 共享一份），找和当前对话相关的记忆文件。

核心价值是**隐藏延迟**——搜记忆要磁盘 I/O，趁模型思考时一起搜（并行），比等它回完再搜（串行）快得多。

几个细节：每轮只消费一次（PTL 重试不重复注入）；去重防止同一记忆文件同会话多次注入；用 `using` 语法保证 generator 退出时（正常/异常/中断）自动清理。

## 3.9 反应式压缩（最后兜底）

正常情况下 autocompact 应该在 ~83%~90% 主动触发，把 PTL（Prompt-Too-Long，提示词超长）错误挡在门外。但有些情况挡不住：

- autocompact 被禁用或跳过
- 单次工具结果异常大，一步跨过阈值
- Context Collapse 排水不够

我们的登录任务就撞上一次：某轮一个工具结果炸到 30 万 token，直接触发 PTL。这时反应式压缩启动——它没有 autocompact 的"主动预防"从容，是**已经出事了**才上的兜底。

机制：按 API 轮次分组，从消息头部丢弃最旧的轮次来缩小请求，最多重试 3 次。

> 两个相关设计决策：
> - **压缩阈值怎么定的**：`tokens >= effectiveWindow - 13000`，`effectiveWindow = 上下文窗口 - min(模型输出上限, 2万)`。13K buffer 保证触发时还有空间跑完当前工具和生成摘要。警告阈值再提前 2 万 token，让用户在被压缩前就看到提示。
> - **max_output_tokens 默认为何只用 8 千、不用 3.2 万**：统计上 99% 请求输出不到 5 千，声明 3.2 万会让服务端过度预留资源。不够时自动升到 6.4 万重试（MOT 恢复）。

## 3.10 实践指南：怎么用更省更快

理解了上面这些，反过来想：你日常哪些习惯能最大化缓存命中（更快、更省），哪些会无意"打碎"缓存？

**1. 别中途断太久**。普通用户缓存 TTL 是 **5 分钟**（付费 1 小时）。离开超 5 分钟回来，整个前缀从头算，第一轮明显变慢。
> 要短暂离开思考，尽量 5 分钟内回来。回不来也别慌——第一轮慢是 TTL 过期的正常现象，后面立刻恢复。

**2. 相关任务用长对话，别频繁新建会话**。每次新会话都是完全冷启动，5~10 万 token 的系统提示词和工具定义从头算。同一会话继续，这些都在缓存里，每轮只算新增。
> 同时开多个 Claude Code 会话，它们之间的消息历史缓存是独立的（系统提示词部分能跨会话共享）。

**3. 让自动压缩替你管**。五级流水线会在接近窗口时自动触发，你不用手动干预。
> 看到使用率警告别紧张，让系统处理。只有你想"重新开始一段相关逻辑"时才手动 `/compact`，逻辑可以直接 `/clear`。

**4. 精简 MCP 工具安装**。**只要 MCP 工具实际渲染进了工具数组**（没被延迟加载），系统提示词缓存就从 global 降级到 org——你享受不到全球共享缓存，每次冷启动独立算。
> 只装真正在用的 MCP server。偶尔用一次的，用完移除。MCP 越少缓存越高效。（被 Tool Search 延迟加载的 MCP 工具不触发降级。）

**5. 别频繁切模型**。不同模型在服务端用不同 KV Cache 空间，同会话切来切去每次都命中不了之前的缓存。
> 一个会话尽量用同一个模型。要切，不如开新会话。

**心智模型**收尾：

```
你的每次请求 = [已缓存的前缀] + [新增的内容]
                ~~~~~~~~~~~~     ~~~~~~~~~~~~
                免费（已有 KV Cache）  要算（花时间花钱）
```

你的目标是最大化"已缓存的前缀"。核心原则就一条——**保持稳定**：会话连续、少重置、少做改变前缀的操作。前缀越稳，命中越高，响应越快，成本越低。

## 3.11 设计洞察

回到我们的登录任务收个尾，这章几个最值得记住的设计取舍：

1. **Memoize 保证幂等**：git 状态、CLAUDE.md 每会话只算一次，不重复劳动。
2. **压缩渐进**：从零成本裁剪到全量摘要逐级升级，大部分对话永远不会触发 Autocompact。
3. **折叠可逆**：Context Collapse 不动原件、能回退——这是它优于 Autocompact 的根本。
4. **缓存感知的组装顺序**：最稳的系统提示词在最前，最利于命中。
5. **锚点估算**：用服务端精确值当锚、只估增量，在精度和延迟间取平衡。
6. **system-reminder 统一通道**：XML 标签在任意位置注入系统信息，不混淆角色边界。
7. **会话级锁存的务实取舍**：TTL 资格和 beta header 一旦确定就锁到会话结束，牺牲灵活性换缓存稳定——10 万 token 缓存失效的代价远高于中途关不掉某个功能。

整章一句话：上下文工程是在"给模型最好的视野"和"不破坏缓存"两股力之间做平衡的一门手艺。你看到的每一个看似过度复杂的机制，拆开看都是在解这个约束。
